# WZD 기획 보고서

## 1. 지금 잠가야 할 제품 정의
지금 WZD는 아래 한 문장으로 잠그는 것이 맞다.

**WZD는 브라우저를 열면 가장 먼저 보고 싶은 개인화 첫 페이지다.**

보조 설명은 다음 정도로 제한한다.
- 자주 가는 북마크를 정리할 수 있다
- 관심 있는 RSS를 한 화면에 모아볼 수 있다
- 개인 취향대로 꾸미고 배치할 수 있다
- 필요하면 외부에 공유 가능한 개인화 페이지로 확장할 수 있다

즉, 현재 메인 포지션은 `메모 보드`도 아니고 `AI 작업 허브`도 아니며 `개인화 첫 페이지`다.

## 2. 문서와 랜딩에서 가장 먼저 바꿔야 할 것
가장 먼저 바꿔야 하는 것은 기능이 아니라 메시지다.

### 먼저 바꿔야 할 축
1. README의 제품 소개 문장
2. 공모전 제출 문서의 제목/한 줄 정의/문제 정의
3. 랜딩 첫 화면의 헤드라인과 보조 카피

### 바꿔야 하는 방향
- `AI 에이전트 기반 작업 보드` 톤 제거
- `메모 보드` 중심 설명 제거
- `개인화 첫 페이지 + 북마크 + RSS` 중심으로 재정렬
- 윈도우/모바일 위젯, 동기화, 하드웨어는 확장성으로만 뒤에 둠

## 3. 지금 미뤄야 할 것
현재 라운드에서 미뤄야 하는 것은 아래다.
- 에이전트 기능 전면화
- 마켓 전면화
- 새로운 위젯 확장
- 팀 협업 기능 강조
- 윈도우 위젯/안드로이드 위젯 실개발
- 하드웨어 방향 전개
- 메모 고도화 중심의 서사

이 요소들은 지금 코어를 흐리게 한다.

## 4. 다음 구현 라운드 목표
다음 라운드 목표는 기능 추가가 아니라 `제품 포지셔닝 정렬`이다.

### 다음 구현 라운드 Goal
**WZD가 개인화 첫 페이지로 읽히도록 주요 문서와 랜딩 메시지를 정렬한다.**

이 라운드에서 사용자가 느껴야 하는 변화는 다음이다.
- 이 제품이 무엇인지 한 번에 이해된다
- 북마크와 RSS가 핵심이라는 것이 보인다
- 메모보드/에이전트 허브처럼 보이던 인상이 줄어든다

## 5. 첫 번째로 바뀔 가능성이 높은 파일
우선순위대로 보면 다음 파일들이 1차 수정 대상이다.

1. `README.md`
2. `IDEA_CONTEST_SUBMISSION_WZD.md`
3. `src/features/landing/LandingPage.tsx`
4. 필요 시 `PRODUCT_PLAN.md`
5. 필요 시 `PRODUCT_EXECUTION.md`

현재 라운드에서는 가급적 위 파일에만 집중하는 것이 좋다.

## 6. 개발자용 명확한 완료 기준
다음 라운드에서 개발자(Codex)가 만족해야 하는 기준은 아래다.

### Acceptance Criteria
- README 첫 제품 설명이 `개인화 첫 페이지` 중심으로 바뀐다.
- 공모전 문서에서 메인 포지션이 `개인화 첫 페이지 + 북마크 + RSS`로 정리된다.
- 랜딩 첫 화면에서 사용자가 WZD를 `브라우저 첫 페이지`로 이해할 수 있다.
- 랜딩에서 AI 에이전트/작업 허브/과도한 위젯 확장 인상이 메인 메시지를 덮지 않는다.
- `npm run build`가 통과한다.
- UI 변경이 있으면 `/gstack-review`와 `/gstack-qa-only` 기준에서 막히지 않는다.

## 7. 이번 라운드 개발자에게 주는 제약
이번 라운드에서 개발자는 아래를 하면 안 된다.
- 새 기능 추가
- 새 위젯 추가
- 라우팅 구조 변경
- 마켓 기능 손대기
- 인증/저장 구조 변경
- 공모전 메시지와 무관한 UI 정리 작업 확장

이번 라운드는 `포지셔닝 정렬 라운드`다. 구현 범위를 넓히면 실패할 가능성이 크다.

## 최종 정리
지금 WZD에 필요한 것은 더 많은 기능이 아니라 더 선명한 얼굴이다.

이번 라운드는 아래 한 문장만 끝까지 밀면 된다.

**WZD는 브라우저를 열면 가장 먼저 보고 싶은 개인화 첫 페이지다.**
