# WZD 검증 보고서

## 1. 현재 제품 정의
현재 WZD는 코드 기준으로 보면 단순한 시작페이지가 아니라 다음이 한꺼번에 섞여 있는 상태다.
- 개인 보드 / 메모 보드
- 북마크 / RSS / 체크리스트 / 디데이 / 시간표 / 날씨 / 트렌드 등 다양한 위젯 시스템
- 공유 보드 / 마켓 / 템플릿
- 에이전트 실행 개념
- 랜딩 페이지에서는 생산성 보드/작업 보드 톤이 강함

즉 현재 제품은 `개인화 첫 페이지`라기보다 `메모 보드 + 위젯 보드 + 협업 보드 + 에이전트 허브`에 가깝다.

## 2. 지금 너무 넓은 부분
현재 가장 큰 문제는 제품 정의가 지나치게 넓다는 점이다.
- 시작페이지인지
- 메모 보드인지
- 생산성 보드인지
- 협업 보드인지
- 에이전트 툴인지
- 위젯 마켓 플랫폼인지
한 번에 다 하려는 인상이 있다.

이 상태에서는 사용자도, 심사위원도, 개발자도 제품을 한 문장으로 이해하기 어렵다.

## 3. 지금 잡아야 할 코어 웨지
지금 WZD가 가장 강하게 잡아야 할 코어는 아래다.

### 코어 웨지
**브라우저를 열면 가장 먼저 보고 싶은 개인화 첫 페이지**

이 첫 페이지의 핵심 기능은 다음 두 가지로 좁혀야 한다.
1. 북마크
2. RSS

메모는 보조 기능으로 둘 수 있지만 메인 포지션이 되어서는 안 된다.
링크페이지 기능, 윈도우 위젯, 모바일 위젯, 동기화, 마켓, 에이전트는 전부 후순위다.

## 4. 첫 번째 타깃 유저
가장 먼저 잡아야 할 사용자는 아래다.

### 1차 타깃
- 브라우저 중심으로 일하는 사람
- 탭을 많이 열어두는 사람
- 자주 보는 링크와 읽을거리를 한 화면에서 정리하고 싶은 사람
- AI/개발/콘텐츠/리서치처럼 정보 소비량이 많은 사용자

즉 대중 전체보다 먼저 `브라우저 헤비유저`에 맞춰야 한다.

## 5. MVP는 무엇이어야 하는가
지금 MVP는 아래 정도로 충분하다.

### MVP
- 북마크를 쉽게 추가/정리/배치할 수 있다
- RSS 피드를 쉽게 붙이고 읽을 수 있다
- 페이지를 개인화해서 첫 화면처럼 쓸 수 있다
- 보기 좋고 빠르며, 매일 열어도 부담이 없다

### MVP에서 제외하거나 약하게 둘 것
- 고급 메모 보드 확장
- 에이전트 실행
- 마켓
- 과도한 협업 기능
- 위젯 종류 확장 경쟁
- 윈도우/모바일 위젯 연동
- 하드웨어 비전

## 6. 지금 잘라내거나 미뤄야 할 것
아래는 당장 미루는 게 맞다.
- 에이전트 허브 포지션
- 마켓 전면화
- 너무 많은 위젯 종류 강조
- 협업/초대/공유 기능 중심 설명
- 메모 보드 중심 포지션
- 윈도우/모바일 위젯 실개발
- 하드웨어 이야기의 전면 배치

이 요소들은 방향을 흐린다. 확장성 문서에는 남길 수 있지만 지금 메인 메시지에서는 빼야 한다.

## 7. 다음으로 검증해야 할 것
다음 라운드에서 검증해야 하는 것은 기능 추가가 아니라 메시지와 핵심 경험이다.

### 바로 검증할 질문
1. 사용자가 WZD를 보고 `아, 이건 내 브라우저 첫 페이지구나`라고 이해하는가?
2. 북마크 + RSS만으로도 충분히 다시 열고 싶은 이유가 생기는가?
3. 랜딩 문구가 생산성 보드가 아니라 개인화 첫 페이지로 읽히는가?
4. 링크페이지 대체/공유 기능은 코어 이후의 확장으로 자연스럽게 보이는가?

## 최종 결론
지금 WZD는 기능이 부족한 게 아니라 정의가 넓다.

### 지금 해야 할 일
- 제품 정의를 `개인화 첫 페이지`로 고정
- 코어 기능을 `북마크 + RSS`로 압축
- 랜딩/README/공모전 문서를 이 방향으로 통일
- 나머지는 확장으로 뒤로 미룸

지금 필요한 것은 더 많은 기능이 아니라 **더 날카로운 포지셔닝**이다.
