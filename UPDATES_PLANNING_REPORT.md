# WZD 업데이트 게시판 기획 보고서

## 1. 지금 잠가야 할 정의
이번 기능은 `공지사항 게시판`이 아니라, WZD의 제품 변경 이력과 방향을 검색엔진과 사용자에게 축적하는 **업데이트 허브**로 정의하는 것이 맞다.

### 한 문장 정의
**WZD 업데이트 허브는 제품 변경 내용을 상세하게 기록하고 검색 가능한 자산으로 축적하는 작은 SEO/신뢰 레이어다.**

## 2. 이번 라운드 목표
이번 라운드 목표는 게시판을 크게 만드는 것이 아니라, 정적인 데이터 기반으로 아래를 먼저 성립시키는 것이다.

- `/updates` 목록 페이지
- `/updates/:slug` 상세 페이지
- 2~3개의 실제 업데이트 글 시드 데이터
- 제목, 요약, 본문, 날짜, 태그, SEO 메타 필드 구조

즉 이번 라운드의 본질은 `게시판 시스템 구축`이 아니라 **업데이트 콘텐츠를 제품 안에 노출하는 구조 만들기**다.

## 3. 왜 지금 이 범위가 맞는가
- 이미 쓸 수 있는 실제 업데이트 소재가 있다.
  - 개인화 첫 페이지 포지셔닝 정리
  - 랜딩 개편
  - Insight Reader MVP 추가
- WZD는 초기 제품이므로, 완성도보다 `계속 움직이고 있다`는 인상이 중요하다.
- CMS 없이도 정적 데이터 기반으로 충분히 시작할 수 있다.

## 4. 이번 라운드 In scope
- 업데이트 글 데이터 파일 추가
- `/updates` 목록 UI 추가
- `/updates/:slug` 상세 UI 추가
- 글별 기본 SEO title / description 연결 가능 구조 추가
- 랜딩 또는 관련 표면에서 업데이트 허브로 들어가는 최소 링크 추가

## 5. 이번 라운드 Out of scope
- 관리자 CMS
- 글 작성 UI
- 댓글
- 좋아요/조회수
- 검색/필터링 고도화
- RSS 피드 발행
- 뉴스레터 연결
- 마크다운 에디터 구축
- 서버 측 콘텐츠 관리 기능

## 6. 첫 글 시드 추천
### 글 1
- 제목: WZD 업데이트: 개인화 첫 페이지 방향으로 재정의
- 핵심: 메모보드/허브가 아니라 브라우저 첫 페이지로 포지셔닝 전환

### 글 2
- 제목: WZD 업데이트: 랜딩 페이지를 북마크와 RSS 중심으로 개편
- 핵심: 랜딩 메시지와 첫인상 조정

### 글 3
- 제목: WZD 업데이트: Insight Reader MVP 추가
- 핵심: Supabase `get_home_payload` 기반 최소 인사이트 표면 추가

## 7. 우선 변경될 가능성이 높은 파일
- `src/App.tsx`
- `src/features/updates/UpdatesIndexPage.tsx`
- `src/features/updates/UpdateDetailPage.tsx`
- `src/lib/updates-content.ts`
- `src/styles/main.css`
- 필요 시 `src/features/landing/LandingPage.tsx`

## 8. 개발자 완료 기준
- `/updates`에서 글 목록이 보인다.
- `/updates/:slug`에서 상세 글이 보인다.
- 최소 2~3개의 상세한 업데이트 글이 실제로 들어 있다.
- 제품 포지셔닝과 관련된 키워드가 본문/제목에 자연스럽게 포함된다.
- build가 통과한다.
- 기능이 본 제품보다 앞에 나서지 않고 보조 허브처럼 보인다.

## 9. 개발자 제약
- 이번 라운드는 정적 데이터 기반으로 끝낸다.
- CMS나 입력 시스템으로 확장하지 않는다.
- 업데이트 허브가 메인 제품처럼 보이지 않게 한다.
- 랜딩/본 제품의 큰 구조는 흔들지 않는다.

## 최종 정리
지금 업데이트 게시판은 `작은 기능`이 아니라, WZD를 검색엔진과 사용자에게 살아 있는 제품으로 보이게 하는 축적 장치다.

하지만 이 가치는 `작게 시작할 때`만 살아난다.
이번 라운드는 반드시 **정적 콘텐츠 기반의 작은 업데이트 허브**로 끝내는 것이 맞다.
